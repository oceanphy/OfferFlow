# 0003 — 证据源抽象：知识库不硬编码为单一来源

诊断所需的参考知识不限定为"八股知识库"，而是抽象为证据源（Evidence Source）接口。当前只实现知识库条目，但接口预留了简历快照、代码仓库等扩展点。

**Considered Options**

1. **硬编码知识库** — 简单直接，表结构一次定死。但面试诊断经常需要结合简历内容判断回答质量（"你项目里用了 Redis"但代码里其实没用），纯八股知识库无法覆盖。
2. **证据源抽象** — 多一层间接，当前只有一种实现但引入了接口复杂度。换来的是后续可以按需挂载不同知识来源，且诊断 sub-agent 不需要关心来源细节。

选择方案 2。用户明确提出"知识库来源可能很广泛"，需要在设计时预留空间。接口层的额外成本很低（本质就是一个 get_context(query) → str），但后期从硬编码迁移到多来源的成本很高。

**Consequences**

- 当前阶段只有 KnowledgeBaseEntry 一种实现，接口可能过度设计——如果后续一直没有第二种实现出现，应考虑移除抽象。
- 不同证据源的检索方式差异大（知识库用语义检索、代码仓库用符号搜索），抽象接口需要保持足够宽泛。
