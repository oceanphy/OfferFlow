# 面试诊断 Agent

面试准备最大的问题不是“不知道答案”，而是“不知道自己答得有什么问题”。

候选人花大量时间刷题，但真正上场时暴露的问题往往不是知识盲区——而是表达混乱、逻辑断层、关键点遗漏。这些问题自己很难察觉，复盘时也抓不到重点。

本项目要做的就是：一个能精准诊断面试回答的 Agent 系统。

## 产品定位

**面试诊断 Agent** —— 上传面试文字稿，自动诊断回答质量，输出结构化改进建议。

核心价值：

- 不只是给答案，而是告诉你“你的回答哪里有问题、为什么有问题、怎么改”
- 不只看内容，还看表达：逻辑连贯性
- 背后自己总结的 Agent 知识库，诊断标准不是拍脑袋，是工程化的评估维度

## 用户场景

场景：文字稿诊断

用户上传面试文字稿——>agent识别每道题的回答——>逐题诊断——>输出改进建议。

## 系统架构：基于 Harness 工程的 10 层设计

核心设计决策：**不用框架，手写 Harness。** Agent loop、tool dispatch、stream 解析、context 管理全部自己实现。代码即教程，每一层都是可独立讲解的工程模块。

## 第 1 层：Tools —— 原子能力清单

| `split_qa_pairs`  | 从连续文本中拆分出 Q&A 对 |
| ----------------- | ------------------------- |
| `analyze_content` | 诊断回答的内容质量        |

| `query_knowledge_base` | 从知识库检索相关高手答 |
| ---------------------- | ---------------------- |
|                        |                        |

| `generate_report` | 汇总诊断结果生成报告 |
| ----------------- | -------------------- |
| `read_file`       | 读取用户上传的文件   |

设计原则：

- 每个 Tool 只做一件事，参数 schema 严格定义
- 返回结构化 JSON，不是自由文本
- 错误分类明确：`input_error` / `service_error` / `timeout`
- 所有调用记录审计日志

## 第 2 层：Skills —— 任务级能力

Skills 是 Tools 的有意义组合，代表一个完整的诊断流程。

**核心 Skills：**触发用户上传文字稿/粘贴内容——>`split_qa_pairs`——>`query_knowledge_base`——>`analyze_content`——>`generate_report`

## 第 3 层：Query Engine —— 模型调用的工程化

关键设计：

- **流式输出**：诊断过程实时显示，不让用户干等
- **缓存**：相同问题的知识库检索结果缓存，避免重复计算
- **Token 预算**：每次诊断设定 token 上限，防止成本失控
- **降级**：主模型不可用时，切换备用模型

## 第 4 层：Context —— 信息密度管理

面试诊断的上下文特别容易爆：一场面试可能有 20+ 道题，每题的回答、参考答案、诊断结果都在膨胀。

分层注入：

system prompt：诊断标准+评分维度（长度固定）	系统提示词：小于2000tokens

current Task：当前正在诊断的Q&A对（动态）	当前任务上下文：小于2000tokens

reference：该题对应的知识库参考（按需加载）	参考资料：小于3000tokens

history：已完成诊断的摘要（压缩后保留）	历史摘要：小于2000tokens

## 第 5 层：Memory —— 用户画像与诊断历史

user-profile（长期）：用户目标岗位、技术栈背景、历次诊断中暴露的共性弱点、改进趋势

diagnosis-history（中中期）：每次诊断的结构化摘要、高频失分维度统计、进步/退步对比

session-context（短期）：当前诊断会话工作状态、已处理/待处理的Q&A队列

Memory 写入规则（克制原则）：

- 只保存稳定结论，不保存中间推理过程
- 只保存用户画像变化，不保存每次对话细节
- 诊断结果只保存摘要 + 关键数据点

## 第 6 层：Permission —— 安全与隐私底座

面试内容是高度隐私数据，权限设计必须严格：

隐私保护：

- 用户可随时要求删除所有诊断历史
- 模型调用不附带用户身份信息

## 第 7 层：Sessions —— 诊断过程可恢复

一场面试诊断可能需要 10–30 分钟。Session 必须支持中断恢复。

## 第 10 层：Sub-agent —— 诊断任务的并行分发

单题诊断可以串行，但多题诊断适合并行分发：

Sub-agent 职责划分：

| Agent    | 职责                           | 可用 Tools                            | 上下文边界            |
| -------- | ------------------------------ | ------------------------------------- | --------------------- |
| 内容诊断 | 评估回答的完整性、深度、准确性 | query_knowledge_base, analyze_content | 单题 Q&A + 参考答案   |
| 表达诊断 | 评估逻辑结构、措辞、条理       | analyze_content                       | 单题回答文本          |
| 知识对标 | 对比高手答，输出差距           | query_knowledge_base                  | 单题 Q&A + 知识库全文 |
| 报告生成 | 汇总所有诊断，生成最终报告     | generate_report                       | 所有诊断摘要          |
