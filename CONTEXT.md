# OfferFlow — 面试诊断系统

对面试文字稿进行结构化诊断：定位回答中的问题、分析原因、给出可执行的改进建议。

## 核心实体

**面试文字稿（Interview Transcript）**:
用户上传的面试对话文本。当前支持事后整理稿，后续可扩展到实时语音转写。
_Avoid_: 录音、语音稿、逐字稿

**面试回合（Interview Round）**:
一个话题加上可能的追问链，是诊断的最小单元。一场面试包含多个回合。回合内可能包含多轮一问一答（追问），但诊断时作为一个整体处理。
_Avoid_: Q&A 对、问答对

**诊断（Diagnosis）**:
对单个面试回合的三步分析过程：问题定位 → 原因分析 → 改进建议。区别于只给分数不深入原因的"评测"和逐题过流程的"复盘"。
_Avoid_: 评测（Assessment）、复盘（Review）、打分

**诊断报告（Diagnosis Report）**:
一场面试的完整诊断输出，包含所有回合的诊断结果汇总。

## 诊断报告结构

**问题识别（Question Recognition）**:
从原始文字稿中提取并重述面试官的问题，补充隐含的考察意图。

**回答复述（Answer Restatement）**:
对候选人回答的去噪声版本，保留原意但清除口语化杂质。

**评分（Scoring）**:
按维度对回答打分。由内容维度和表达维度组成，当前比例为 60/40，开发者可调。

**评分维度（Scoring Dimension）**:
评分体系下的子项。当前分为：

- **内容维度（Content Dimension）** — 占比 60%，含完整性、准确性、深度
- **表达维度（Expression Dimension）** — 占比 40%，含逻辑连贯性、结构性、措辞精准度

**考察维度（Evaluation Dimension）**:
面试官通过该题实际想考察的能力点。一道题可能考察多个维度。区别于评分维度——评分维度是通用的打分框架，考察维度是具体到某道题的考察意图。

**亮点（Highlight）**:
回答中值得肯定的具体片段，附带原文引用。

**差距（Gap）**:
回答中缺失或不足的部分，定位到具体问题点，对比参考标准说明为什么不足。

**改进建议（Improvement Suggestion）**:
可执行的修改方向。每条建议定位到具体回答片段 + 引用参考标准 + 给出修改方向。
_Avoid_: 建议、意见、改进方向

**参考回答（Model Answer）**:
针对该问题的标准回答示例，分完整版和精简版。

**可能追问（Follow-up Question）**:
面试官基于当前回答可能继续追问的问题列表。

## 证据源

**证据源（Evidence Source）**:
诊断时按需拉取的知识来源的抽象。当前实现从知识库条目检索，模型预留扩展接口。

**知识库条目（Knowledge Base Entry）**:
针对特定面试题目的参考答案、评分标准和常见错误的结构化条目。来源为用户自维护的八股积累。

**简历快照（Resume Snapshot）**:
候选人简历的原文片段。当诊断需要结合简历内容时按需加载。预留。

**代码仓库（Code Repository）**:
候选人的项目代码。当诊断需要结合具体实现时按需加载。预留。

## Sub-agent 体系

诊断任务按职责分发到以下 sub-agent，每个 sub-agent 拥有独立的上下文边界和可用 tool：

| Sub-agent | 职责 | 上下文边界 |
|---|---|---|
| **内容诊断（Content Diagnosis Agent）** | 评估回答的完整性、深度、准确性 | 单回合 + 知识库条目 |
| **表达诊断（Expression Diagnosis Agent）** | 评估逻辑结构、措辞、条理 | 单回合回答文本 |
| **知识对标（Knowledge Benchmarking Agent）** | 对比知识库条目，输出差距 | 单回合 + 知识库条目 |
| **报告生成（Report Generation Agent）** | 汇总所有诊断，生成最终报告 | 所有诊断摘要 |
